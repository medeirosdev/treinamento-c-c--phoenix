Introdução e Configuração

5 questões

O que é Controle de Versão?

Um Sistema de Controle de Versão (VCS) registra alterações em arquivos ao longo do tempo, permitindo recuperar versões específicas, comparar mudanças, identificar quem introduziu um bug e quando — e reverter o projeto inteiro para um estado anterior caso algo dê errado.

Três gerações de VCS

TipoExemploCaracterística
LocalRCSBanco de dados de patches numa única máquina
Centralizado (CVCS)SVN, PerforceServidor único; se cair, ninguém trabalha
Distribuído (DVCS)Git, MercurialCada cliente possui o repositório completo com todo o histórico
Por que distribuído é melhor? Se o servidor morrer, qualquer clone de cliente pode restaurá-lo. Todo trabalho offline é possível — você commita localmente e sincroniza depois.

Uma Breve História do Git

Em 2002 o kernel Linux usava o sistema proprietário BitKeeper. Em 2005 o relacionamento entre a comunidade e a empresa se rompeu. Linus Torvalds então criou o Git com objetivos claros:

  • Velocidade
  • Design simples
  • Forte suporte a desenvolvimento não-linear (milhares de branches paralelos)
  • Totalmente distribuído
  • Capaz de lidar com projetos grandes como o kernel Linux

Como o Git Pensa nos Dados

A maioria dos outros VCS armazena dados como uma lista de mudanças por arquivo (delta-based). O Git é diferente: ele pensa nos dados como uma série de snapshots de um mini sistema de arquivos.

Outros VCS (delta-based): Arquivo A ──── Δ1 ─────────────── Δ2 Arquivo B ──────────── Δ1 ──── Δ2 Versão: V1 V2 V3 V4 Git (snapshots): V1 V2 V3 V4 A1 A1─────── A2 A2 B (─────) B1 B1 B2 C1 C2 C2 (────) C3 (dashed = link para o snapshot anterior idêntico)

Instalação do Git

$ sudo apt install git-all # Debian/Ubuntu $ sudo dnf install git-all # Fedora/RHEL $ git --version # verificar instalação git version 2.45.0

Configuração Inicial — git config

Configurações são armazenadas em 3 níveis (cada nível sobrescreve o anterior):

  • /etc/gitconfig — sistema (--system)
  • ~/.gitconfig — usuário (--global)
  • .git/config — repositório (--local)
$ git config --global user.name "Seu Nome" $ git config --global user.email "voce@email.com" $ git config --global core.editor vim $ git config --global init.defaultBranch main $ git config --list # ver todas as configurações
Dica: Use git config --global apenas uma vez por máquina. O nome e e-mail são gravados em cada commit que você criar — escolha com cuidado.

Fundamentos — init, add, commit, log

5 questões

Obtendo um Repositório

Há duas formas de ter um repositório Git local:

# 1. Inicializar em diretório existente $ cd meu-projeto $ git init Initialized empty Git repository in /meu-projeto/.git/ # 2. Clonar repositório existente $ git clone https://github.com/usuario/repo.git $ git clone https://github.com/usuario/repo.git meu-nome # nome personalizado

Ciclo de Vida dos Arquivos

Arquivos no repositório vivem entre dois mundos: rastreados (tracked) e não-rastreados (untracked).

Untracked Unmodified Modified Staged │ │ │ │ │── git add ──────────────────────────▶ │ │ │── editar ──▶ │ │ │ │ │── git add ▶│ │ │ │ │── git commit ──▶ (Unmodified) │ ◀─────── git rm ───────────────────── │

Verificando o Status

$ git status On branch main Your branch is up to date with 'origin/main'. nothing to commit, working tree clean $ echo "Phoenix" > README.md $ git status Untracked files: (use "git add <file>..." to include in what will be committed) README.md $ git status -s # versão compacta ?? README.md

Legenda do git status -s

CódigoSignificado
??Arquivo não rastreado (untracked)
ANovo arquivo adicionado ao stage
MArquivo modificado
MMModificado, staged e depois modificado novamente

Adicionando Arquivos — git add

$ git add README.md # arquivo específico $ git add src/ # diretório inteiro (recursivo) $ git add *.c # padrão glob $ git add -A # tudo (novos + modificados + removidos)
Dica importante: git add captura o arquivo no momento em que é executado. Se você modificar o arquivo depois do git add, precisará rodá-lo novamente para incluir a nova versão no commit.

Ignorando Arquivos — .gitignore

Crie um arquivo .gitignore na raiz do projeto antes do primeiro commit:

# .gitignore *.o # todos os .o (objetos compilados) *.a # arquivos de biblioteca !lib.a # exceto lib.a (nega a regra anterior) /TODO # apenas o TODO na raiz (não em subdiretórios) build/ # diretório build inteiro doc/*.txt # .txt em doc/ mas não em doc/sub/ doc/**/*.pdf # todos os .pdf em qualquer nível abaixo de doc/

Visualizando Diferenças — git diff

$ git diff # mudanças não staged (working tree vs stage) $ git diff --staged # mudanças staged (stage vs último commit) $ git diff HEAD~1 # comparar com commit anterior

Criando um Commit

$ git commit -m "Adiciona arquivo README" [main (root-commit) 3a8f2b1] Adiciona arquivo README 1 file changed, 1 insertion(+) $ git commit -a -m "Atualiza README" # -a faz add de todos os tracked automaticamente
Anatomia de um commit: hash SHA-1 (40 chars), autor, data, mensagem, ponteiro para o tree (snapshot), ponteiro para o(s) commit(s) pai.

Visualizando o Histórico — git log

$ git log # histórico completo $ git log --oneline # resumido, 1 linha por commit $ git log --oneline --graph # com grafo ASCII de branches $ git log --oneline --graph --all # todas as branches $ git log -5 # últimos 5 commits $ git log --since="2 weeks ago" # filtro por data $ git log --author="Medeiros" # filtro por autor $ git log --grep="fix" # filtro por mensagem

Removendo e Movendo Arquivos

$ git rm arquivo.txt # remove do stage E do disco $ git rm --cached arquivo.txt # remove só do rastreamento (mantém no disco) $ git mv antigo.c novo.c # equivale a: mv + git rm + git add

Branches e Merging

5 questões

O que é uma Branch?

Uma branch no Git é simplesmente um ponteiro leve e móvel para um commit. Criar uma branch é escrever 41 bytes num arquivo — é instantâneo, independentemente do tamanho do projeto.

HEAD — Onde Você Está

HEAD é um ponteiro especial que indica em qual branch (ou commit) você está trabalhando. Quando você faz um commit, a branch atual avança automaticamente.

C0 ←── C1 ←── C2 ←── C3 ↑ main ← HEAD

Criando e Trocando de Branch

$ git branch feature # cria branch (não muda para ela) $ git checkout feature # muda para a branch $ git checkout -b feature # cria E muda — atalho # Forma moderna (Git ≥ 2.23) $ git switch feature # muda para branch existente $ git switch -c feature # cria e muda $ git switch - # volta para a branch anterior $ git branch # lista branches locais $ git branch -v # com último commit de cada $ git branch --merged # branches já mergeadas em HEAD $ git branch --no-merged # branches ainda não mergeadas

Workflow Típico com Branches

Cenário: você está desenvolvendo a feature #53, recebe um bug crítico e precisa fazer um hotfix imediatamente.

# 1. Criar branch para a feature $ git checkout -b iss53 $ # ... trabalhar ... $ git commit -a -m "Início do footer [issue 53]" # 2. Bug crítico! Voltar para main e criar hotfix $ git checkout main $ git checkout -b hotfix $ git commit -a -m "Corrige e-mail quebrado" # 3. Mesclar hotfix em main (fast-forward) $ git checkout main $ git merge hotfix Updating f42c576..3a0874c Fast-forward index.html | 2 ++ $ git branch -d hotfix # deletar branch desnecessária # 4. Voltar à feature e continuar $ git checkout iss53

Tipos de Merge

Fast-Forward

Ocorre quando a branch que receberá o merge está diretamente atrás da que será mesclada — não há trabalho divergente. O Git simplesmente move o ponteiro para frente.

Antes: main ──▶ C2 / hotfix ──▶ C4 (à frente de C2) Depois: main ──▶ C4 (ponteiro simplesmente avança)

3-Way Merge (Merge Commit)

Quando as histórias divergiram, o Git usa os dois snapshots das pontas e o ancestral comum, criando um merge commit com dois pais.

main ──▶ C4 / C0─C1─C2 \ C3─C5 ◀── iss53 Após git merge iss53 em main: C0─C1─C2─C4─C6 ◀── main (merge commit C6 tem dois pais)

Resolvendo Conflitos

$ git merge iss53 CONFLICT (content): Merge conflict in index.html Automatic merge failed; fix conflicts and then commit the result. # Conteúdo do arquivo em conflito: <<<<<<< HEAD contact@exemplo.com.br ======= suporte@exemplo.com.br >>>>>>> iss53 # Edite o arquivo, resolva o conflito, depois: $ git add index.html $ git commit # Git preenche a mensagem de merge automaticamente
Marcadores de conflito: <<<<<<< HEAD é a versão da sua branch atual; ====== é o separador; >>>>>>> branch é a versão sendo mergeada. Remova os marcadores e escolha o conteúdo correto.

Remotos e Colaboração

5 questões

O que são Remotos?

Repositórios remotos são versões do projeto hospedadas na Internet ou na rede. Você pode ter vários remotos, cada um com permissões de leitura ou escrita.

$ git remote -v # listar remotos origin git@github.com:user/projeto.git (fetch) origin git@github.com:user/projeto.git (push) $ git remote add origin git@github.com:user/projeto.git # adicionar $ git remote rename origin upstream # renomear $ git remote remove upstream # remover $ git remote show origin # inspecionar

Clonando um Repositório

$ git clone https://github.com/user/projeto.git $ git clone git@github.com:user/projeto.git # via SSH

git clone automaticamente: (1) inicializa .git, (2) baixa todos os dados, (3) faz checkout da última versão da branch padrão, e (4) cria o remote origin.

Fetch, Pull e Push

ComandoO que faz
git fetch originBaixa novos dados do remoto sem aplicar no working tree
git pullEquivale a git fetch + git merge da tracking branch
git push origin mainEnvia commits locais da branch main para o remoto
$ git fetch origin # baixa, não aplica $ git merge origin/main # aplica manualmente $ git pull # fetch + merge em um passo $ git pull --rebase # fetch + rebase (histórico mais limpo) $ git push origin main # envia branch main $ git push -u origin main # envia E configura tracking $ git push origin --delete feature # deleta branch remota
Atenção: git push falha se alguém publicou commits que você ainda não tem. Você precisa primeiro fazer git pull (integrar as mudanças), resolver eventuais conflitos, e então git push novamente.

Tracking Branches

Quando você clona ou usa -u ao fazer push, o Git configura uma tracking branch: uma relação entre sua branch local e a remota correspondente.

$ git branch -vv # ver tracking branches * main 3a8f2b1 [origin/main] Adiciona README hotfix 9c8d1a2 [origin/hotfix: ahead 2] Corrige bug $ git checkout --track origin/feature # criar tracking branch local

Primeiro Envio ao GitHub

Fluxo completo para subir um projeto novo:

# 1. Inicializar e fazer primeiro commit $ git init $ git add . $ git commit -m "Initial commit" # 2. Criar repositório no GitHub (via web), depois: $ git remote add origin git@github.com:usuario/repo.git $ git branch -M main $ git push -u origin main

Desfazendo Coisas

5 questões

Emendando o Último Commit

Se você esqueceu um arquivo ou errou a mensagem no último commit:

$ git add arquivo-esquecido.c $ git commit --amend -m "Mensagem corrigida" # O commit anterior é substituído — use apenas se ainda não fez push!
Cuidado com --amend após push: Amend reescreve o histórico. Se o commit já foi publicado num remoto compartilhado, isso força quem clonou a fazer rebase. Evite amend em commits já publicados.

Desfazendo o Stage — git restore

$ git restore --staged main.c # remove do stage, mantém edições # Equivalente antigo (ainda funciona): $ git reset HEAD main.c

Descartando Edições — git restore

$ git restore main.c # DESCARTA edições não staged — IRREVERSÍVEL # Equivalente antigo: $ git checkout -- main.c
Atenção: Mudanças não commitadas no Git podem ser difíceis ou impossíveis de recuperar. Use git restore <arquivo> com cuidado.

git reset — Três Modos

ModoComandoHEAD move?Stage limpo?Working tree limpa?
--softgit reset --soft HEAD~1✓✗✗
--mixed (padrão)git reset HEAD~1✓✓✗
--hardgit reset --hard HEAD~1✓✓✓ (perda!)
# Desfazer último commit, manter arquivos no stage $ git reset --soft HEAD~1 # Desfazer último commit, tirar do stage, manter edições no disco $ git reset HEAD~1 # PERIGO: Desfazer e descartar tudo — voltando ao estado do commit $ git reset --hard HEAD~1

git revert — Desfazer Sem Reescrever Histórico

git revert cria um novo commit que desfaz as mudanças de um commit anterior. É o modo seguro para desfazer em repositórios compartilhados.

$ git revert a1b2c3d # cria commit que desfaz a1b2c3d $ git revert HEAD # desfaz o último commit $ git revert HEAD~3..HEAD # desfaz os últimos 3 commits
reset vs revert: Use reset para desfazer commits locais ainda não publicados. Use revert para desfazer commits já enviados ao remoto compartilhado.

Guardando Mudanças Temporariamente — git stash

Você está no meio de um trabalho, precisa mudar de branch urgentemente, mas não quer commitar algo incompleto.

$ git stash # salva working tree + stage $ git stash push -m "wip: sensor" # com mensagem descritiva $ git stash list # listar stashes stash@{0}: On main: wip: sensor stash@{1}: WIP on feature: a3b4c5 add pid $ git stash pop # restaura o mais recente e remove da lista $ git stash apply stash@{1} # restaura específico sem remover $ git stash drop stash@{1} # remove da lista $ git stash clear # remove todos os stashes

Rebase, Tags e Aliases

5 questões

Rebase — Histórico Linear

O rebase é a segunda forma de integrar branches. Em vez de criar um merge commit, ele reaplica seus commits sobre a ponta da outra branch, criando um histórico linear.

Situação inicial: C0─C1─C2─C4 ◀── main └─C3─C5 ◀── feature Após: git checkout feature && git rebase main C0─C1─C2─C4─C3'─C5' ◀── feature ↑ (C3 e C5 são reescritos com novo hash)
$ git checkout feature $ git rebase main # reaplica commits de feature sobre main $ git checkout main $ git merge feature # agora é fast-forward — histórico linear!

Merge vs Rebase

AspectoMergeRebase
HistóricoPreserva tudo (ramificações visíveis)Linear, mais limpo
HashesCommits originais preservadosCommits reescritos (novos hashes)
SegurançaSeguro em remoto compartilhadoNunca rebase commits já publicados
Uso típicoFeature → main, hotfixAtualizar feature com main antes do PR
A Regra de Ouro do Rebase: Nunca faça rebase de commits que existem fora do seu repositório local. Se alguém já baseou trabalho nos commits que você vai reescrever, haverá problemas graves.

Tags — Marcando Versões

Tags são ponteiros estáticos para commits específicos — usados para marcar releases como v1.0, v2.3.1.

Tags Anotadas (recomendadas)

$ git tag -a v1.0 -m "Versão 1.0 — lançamento inicial" $ git tag -a v1.2 9fceb02 # tag em commit específico (hash parcial) $ git show v1.0 # ver detalhes da tag

Tags Leves

$ git tag v1.4-lw # apenas o checksum do commit

Gerenciando Tags

$ git tag # listar tags $ git tag -l "v1.*" # filtrar por padrão $ git push origin v1.0 # enviar tag específica $ git push origin --tags # enviar todas as tags $ git tag -d v1.4-lw # deletar localmente $ git push origin --delete v1.4-lw # deletar no remoto $ git checkout v2.0.0 # "detached HEAD" — cuidado! $ git checkout -b v2-fix v2.0.0 # criar branch a partir de tag
Dica: Por padrão git push NÃO envia tags. Você precisa fazer isso explicitamente com git push origin --tags ou git push origin <tagname>.

Aliases — Seus Atalhos Personalizados

Configure aliases para comandos que você usa com frequência:

$ git config --global alias.co checkout $ git config --global alias.br branch $ git config --global alias.ci commit $ git config --global alias.st status # Alias para um comando mais complexo $ git config --global alias.unstage 'reset HEAD --' $ git unstage main.c # equivale a: git reset HEAD -- main.c $ git config --global alias.last 'log -1 HEAD' $ git last # mostra o último commit # Alias que chama comando externo (prefixo !) $ git config --global alias.visual '!gitk' # Alias muito útil para histórico visual $ git config --global alias.lg \ 'log --oneline --graph --all --decorate'

Fluxo em Equipe e Boas Práticas

5 questões

Fluxos de Trabalho com Git

O Git não impõe um fluxo único. A comunidade convergiu em alguns padrões:

Feature Branch Workflow

Todo desenvolvimento ocorre em branches isoladas. main (ou master) nunca recebe commits diretos.

# Desenvolvedor começa uma nova feature $ git checkout main $ git pull $ git checkout -b feature/sensor-ultrasonico $ # ... desenvolver ... $ git push -u origin feature/sensor-ultrasonico # Abrir Pull Request no GitHub → code review → merge

Fork & Pull Request (projetos open-source)

  1. Fork do repositório original na sua conta
  2. Clone do seu fork localmente
  3. Crie uma branch, faça as mudanças e commit
  4. Push para o seu fork
  5. Abra um Pull Request do seu fork para o repo original
  6. Dono do repo faz review, discute, e aceita o merge
$ git remote add upstream git@github.com:original/repo.git $ git fetch upstream $ git rebase upstream/main # manter fork atualizado

Boas Mensagens de Commit

Uma boa mensagem de commit é tão importante quanto o código. Facilita o git log, bisect e revisão de código.

Estrutura recomendada:

Linha 1 (assunto): verbo no imperativo, ≤ 72 caracteres
Linha 2: em branco
Linhas 3+: corpo — explica o porquê, não o o quê

✓ Adiciona suporte a sensor HC-SR04
✓ Corrige overflow no acumulador do PID
✗ fix ✗ wip ✗ changes

Comandos para Inspecionar o Histórico

$ git log --oneline --graph --all --decorate * 9e4b2c1 (HEAD -> main) Corrige leitura do encoder * 3a8f2b1 (origin/main) Adiciona PID | * f1d8c2e (feature/sensor) WIP sensor ultrasonico |/ * 7c3d4a5 Initial commit $ git log -S "PIDController" # commits que introduziram/removeram essa string $ git log --follow -p -- motor.c # histórico de um arquivo específico $ git blame motor.c # quem escreveu cada linha $ git show a1b2c3d # detalhes de um commit específico

Bisect — Encontrando o Commit do Bug

Se um bug apareceu em algum momento, git bisect faz busca binária na história:

$ git bisect start $ git bisect bad # HEAD tem o bug $ git bisect good v1.0 # v1.0 estava ok Bisecting: 16 revisions left to test after this (roughly 4 steps) # Git muda para commit do meio — teste manualmente $ git bisect good # ou git bisect bad # Repita até encontrar o commit culpado $ git bisect reset # volta ao HEAD

Cherry-pick — Importar Commit Específico

$ git cherry-pick a1b2c3d # aplica um commit de outra branch na atual
